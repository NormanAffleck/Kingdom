From requirements to prompt engineering to AI-generated code to GitHub Pages deployment — building a working math tool using the complete development workflow.

Project Overview

This activity had us build a general linear equation solver by working through a five-step process: understand requirements, write a prompt, generate with AI, test the output, and host it live on GitHub Pages. The goal wasn't just to get a working tool — it was to learn how to specify what you want precisely enough for AI to build it.

The Five-Step Workflow

1

Understanding the Requirements

Before writing a single line of prompt, we had to answer: what does a "general linear equation solver" actually need to do?

Inputs: coefficients of a system of linear equations · Output: solution values for each variable · Edge cases: no solution (inconsistent), infinite solutions (dependent), single equation, 2×2, 3×3 systems · Format: web-based, static HTML/CSS/JS, no backend required.
2

Build Your Prompt

A well-structured prompt is the entire skill of AI-assisted development. Vague prompts produce vague code.

I am a student building a web tool for a Management Science course. Build a general linear equation solver as a single HTML file (no backend, no dependencies). Requirements: - Support 2×2 and 3×3 systems of linear equations - Allow user to input coefficients and constants - Use Gaussian elimination or Cramer's Rule - Show step-by-step solution process - Handle no-solution and infinite-solutions cases gracefully - Clean, minimal UI — white background, clear typography - Must work offline, hosted on GitHub Pages Output: single .html file only, no explanations needed.
3

Generate with AI

The AI (Claude / ChatGPT) produced a single HTML file implementing Gaussian elimination with partial pivoting. First output worked for standard cases but had issues with singular matrices. We iterated the prompt twice to fix edge case handling.

Key lesson: AI generation is iterative, not one-shot. The first result is a starting point, not the end point. Being specific about edge cases upfront saves iteration cycles.
4

Test

Test cases used to verify the solver:

✅ 2x + y = 5, x − y = 1 → x=2, y=1
✅ 3×3 system with unique solution
✅ Parallel lines (no solution) → correct error message
✅ Same line (infinite solutions) → correct message
✅ Single variable: 3x = 9 → x=3
5

Host on GitHub Pages

Deploy process: create GitHub repo → upload HTML file → Settings → Pages → set branch to main → site goes live at username.github.io/repo-name

GitHub Pages serves static files for free with a public URL. No server, no database, no cost. Perfect for tools built with HTML/CSS/JS.

What Did I Learn?

Knowledge

Gaussian Elimination

Row reduction to upper triangular form, then back-substitution. Works for any n×n system. With partial pivoting, it handles near-singular matrices stably. The same algorithm runs in every computer algebra system.

Skills

Prompt engineering

The quality of AI output is directly proportional to the quality of the prompt. Specifying edge cases, output format, constraints, and purpose upfront produces dramatically better code than "build me a solver."

Reflection

Specification is design

Writing the requirements document was 80% of the work. Once requirements were clear, the AI implementation was fast. This mirrors professional software development: design before code, always.

Branch & Bound

Integer programming's most powerful algorithm — systematically searching a solution space by branching on decision variables and pruning branches that can't beat the current best solution.

Project Overview

Branch & Bound is a technique for solving integer programming problems — where decision variables must be whole numbers (like binary 0/1). Rather than checking every possible solution, it builds a tree of subproblems, uses LP relaxation to compute upper bounds, and prunes any branch where the bound cannot beat the best known feasible solution.

This project involved group whiteboard work on two problems: a general group B&B problem and a B&B application to the pizza toppings problem from Project 2.

How Branch & Bound Works

1

Solve LP Relaxation

Remove integrality constraints. Solve the continuous version — this gives an upper bound on the optimal integer solution.

2

Check for Integrality

If all decision variables in the LP solution are already integers — done! It's the optimal integer solution. If not, pick a fractional variable to branch on.

3

Branch

Create two subproblems: one with the variable forced to ⌊value⌋ and one forced to ⌈value⌉. Each subproblem adds one constraint.

4

Bound & Prune

Solve each subproblem's LP relaxation. If the bound ≤ best known integer solution, prune that branch — it cannot yield an improvement.

5

Update Best Solution

When a node yields an all-integer feasible solution, update the best known solution. Continue until all branches are pruned or explored.

// Key idea: LP relaxation provides UPPER BOUND for maximization
z* (integer) ≤ z_LP (relaxed)

// Pruning conditions
Prune if: z_LP_bound ≤ z_best_integer   (can't improve)
Prune if: LP is infeasible                (no feasible solution in this branch)

Why B&B is More Efficient Than Brute Force

The Pizza Problem has 2¹⁰ = 1,024 possible combinations

Brute force checks all 1,024. Branch & Bound typically explores a small fraction — often under 50 nodes — by pruning branches early. For larger problems (100 variables), this difference is the gap between computing in milliseconds and computing for longer than the age of the universe.

The key insight: you don't need to find the best solution — you need to prove no other solution can be better. Pruning does the proving.

What Did I Learn?

Knowledge

Bounding as proof

The bound isn't just an estimate — it's a proof that an entire subtree cannot contain the optimum. That's what makes B&B exact rather than heuristic. You get the right answer, not just a good one.

Skills

Group whiteboarding

Working through decision trees on Excalidraw as a group made the branching logic visual. Seeing how one bad bound can prune half the tree immediately was much clearer on a whiteboard than in algebra.

Reflection

LP relaxation's role

I originally didn't understand why we "relax" integer constraints first. Now I see it: the relaxed solution is an optimistic estimate — a ceiling. If even that ceiling isn't good enough, you can safely ignore the whole branch.
